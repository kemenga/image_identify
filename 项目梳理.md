# 项目梳理

## 这个项目在做什么

这是一个“文档优先”的图像篡改检测工具，目标不是做通用照片取证，而是优先处理下面几类带文字结构的图片：

- 聊天截图、短信截图里的时间或短文本修改
- 票据、小票里的金额和数字修改
- 身份证、证件里的姓名、号码、地址、照片区域修改
- 发票、票据里少量文字噪声与其余文字噪声不一致的局部修改
- 当文本上下文不足时，退化成局部区域异常检测

项目没有训练模型，也没有深度学习推理流程，核心是传统计算机视觉规则和统计特征。

## 代码结构

### 入口层

- `main.py`
  - 命令行入口
  - 负责读取参数、调用检测器、打印结果、写出 JSON 报告
  - 可通过 `--evidence-output-dir` 额外导出辅助证据热力图

### 融合层

- `detector.py`
  - 暴露统一对外接口 `UniversalTamperDetector`
  - 调用文档检测器拿到候选框
  - 额外计算全局证据：
    - ELA 压缩误差
    - 噪声不一致性
  - 把文档规则分和全局证据融合成最终分数
  - 在评测场景下补充基于噪声证据图的文本行内细化候选
  - 负责结果可视化、辅助热力图导出和报告生成

### 文档规则层

- `document_detector.py`
  - 暴露 `TraditionalTamperDetector`
  - 是项目最核心、也最大的模块
  - 先构建文本行和字符上下文，再生成多类候选区域，最后统一筛选

主要候选来源包括：

- 数字窗口异常：找连续数字串里风格最不一致的字符
- 时间串异常：适合截图里的时间修改
- 短中文异常：适合证件或文档里的短文本篡改
- 文本块热点：从文本密集区里找风格异常区域
- 文字噪声异常：比较少量文字组件和整图/同行文字组件的噪声一致性
- 局部区域异常：滑窗扫描，找和上下文不一致的块
- 证件特化规则：身份证文字区、照片区等定向规则

### 预处理层

- `preprocessing.py`
  - 统一读图与灰度化
  - 检测文本行
  - 在文本行内部切分字符
  - 为上层提供 `CharacterCandidate`

### 测试层

- `test_detector.py`
  - 覆盖截图、票据、身份证三个样例
  - 验证 CLI 输出
  - 验证标准答案抽框、自动对齐和数据集召回
  - 验证报告级误报抑制
  - 验证缺图异常和空白图回退行为

### 评测层

- `evaluation.py`
  - 将 `data/xxx.png` 与 `data/xxx检测结果.png` 自动配对
  - 从标准答案图中抽取红色标注框，尺寸不一致时结合黄色或橙色外层标注稳定框范围
  - 使用 ORB 特征匹配和仿射变换，把尺寸不一致标准答案图中的标注框映射回原图坐标
  - 输出每张图的标准框、检测框、命中情况、召回率、误报数和平均最佳 IoU
  - 默认评测口径是召回优先，`IoU >= 0.3` 即视为命中
  - `400.png` 这类新增变体样例只要补了对应标准答案图，就会自动进入评测

## 核心流程

完整调用链如下：

1. `main.py` 接收图片路径、输出路径、报告路径
2. `detector.py` 通过 `load_image` 读取彩色图和灰度图
3. `document_detector.py` 先检测文本行，再切字符
4. `document_detector.py` 对字符和局部窗口提取多种特征
5. `document_detector.py` 对文字组件做噪声一致性比较，补充“噪声篡改”候选
6. `document_detector.py` 生成候选框并做阈值过滤、去重、合并
7. `detector.py` 计算 ELA 和噪声证据
8. `detector.py` 在大文本块内部扫描噪声图热点，并按文字组件簇细化成多个更小的“噪声篡改”候选
9. `detector.py` 对候选框做二次打分
10. `detector.py` 对最终输出和报告候选做报告级过滤，减少正确文字误判；最终数量语义由 `REPORT_CONFIDENCE_THRESHOLD` 单点控制，专用场景抑制、去重和 `max_detections` 只负责候选合法性、同一区域合并和显式上限
11. `detector.py` 输出高亮图和 JSON 报告
12. 如果传入证据图输出目录，额外生成 ELA、噪声和融合热力图

## 标准答案评测流程

`data/` 目录里名称带 `检测结果` 的图片被视为标准答案图。配对规则是：

- `data/截图.png` 对应 `data/截图检测结果.png`
- `data/票据.png` 对应 `data/票据检测结果.png`
- `data/300.png` 对应 `data/300检测结果.png`
- `data/400.png` 对应 `data/400检测结果.png`
- `data/发票.png` 对应 `data/发票检测结果.png`
- `data/身份证.png` 对应 `data/身份证检测结果.png`

评测命令如下：

```bash
python evaluation.py --data-dir ./data --report ./evaluation_report.json --iou-threshold 0.3 --max-detections 8
```

如果标准答案图和原图尺寸一致，评测优先使用红色标注框，避免外层色块扩大标准框。如果尺寸不一致，评测会先做 ORB 自动对齐，再把红色框与重叠的黄色或橙色外层标注合并，用来减少缩放和半透明标注带来的碎片框。

当前 `data/` 目录里只要存在对应的 `xxx检测结果.png`，就会自动进入量化评测；因此发票样例现在也会参与召回和误报统计。

## 它不是深度学习项目

虽然目录名里有 `deep_learning`，但当前仓库实际没有神经网络训练或推理代码，当前实现完全基于：

- OpenCV
- NumPy
- Pillow
- 手工规则
- 统计离群分数
- 文字组件噪声一致性比较

从现有代码看，这更像一个“文档图像篡改检测规则引擎”。

## 当前输出内容

项目会输出两类结果：

- 标注图：在可疑区域上画框
- JSON 报告：包含状态、检测框、候选框数量、可报告候选数量、Top 候选、全局证据

如果通过 `--evidence-output-dir` 或 Python API 的 `evidence_output_dir` 指定输出目录，还会生成三张辅助证据图：

- `ela_heatmap.png`：ELA 压缩误差热力图
- `noise_heatmap.png`：噪声不一致性热力图
- `fused_heatmap.png`：按 ELA 和噪声权重融合后的热力图

报告中的关键字段有：

- `status`
- `reason`
- `detections`
- `candidate_count`
- `reportable_candidate_count`
- `top_candidates`
- `evidence`
- `evidence_artifacts`

其中 `candidate_regions` 在 Python API 返回值里仍保留完整候选池，适合调试和继续调参；用户可见的 `detections` 和 `top_candidates` 只展示通过报告级过滤的候选。

`REPORT_CONFIDENCE_THRESHOLD` 是最终输出框数量的单一报告级阈值。调高它时，最终 `detections` 和报告 `top_candidates` 的数量只能减少或保持不变；调低它时，数量只能增加或保持不变。设为 `0.0` 表示不按报告级置信度剔除候选，最终数量只应继续受同一物理区域去重、`max_detections` 和 `MAX_REPORT_CANDIDATES` 这些显式上限影响，不能再被隐藏的类型限额压成固定一两个框。

当显式传入该阈值时，融合层还会启用“高阈值保真”排序：高阈值下优先保留截图 `time_group`、票据 `digit_window`、身份证 `*_precise` / `embedded_id_photo`、发票 `invoice_*` 和短信真实 `text_noise_anomaly`，避免让纯粹分值高但语义较弱的小碎片或冲突噪声框排到前面。

高阈值还必须满足“保真”语义：过滤后留下的不是单纯分数最高的原始候选，而是更可信、更接近标准答案的正确候选。当前回归口径明确要求截图保留命中标准答案的 `time_group`、票据保留命中标准答案的 `digit_window`、`300.png` 和 `400.png` 保留命中标准答案的 `text_noise_anomaly`，并且高阈值留下的可见框不应与标准答案完全无重叠。

这条回归不只看候选类型，还会复用 `evaluation.py` 的标准答案抽框和 `bbox_iou` 计算：每个高阈值输出框的最佳 IoU 必须达到 `0.3`，每个标准答案框也必须至少被一个高阈值输出框命中。这样可以防止“类型看起来正确，但框落在错误位置”的退化。

当前 `main.py` 已经把融合层参数和文档规则层参数都接成了 CLI 选项。实际调参时可以直接运行 `python main.py --help` 查看完整参数列表，再通过 `--detector-*` 和 `--document-*` 覆盖默认值，不需要反复改源码。

## 现有样例说明

仓库里自带了 3 张主要样例图：

- `截图.png`
- `票据.png`
- `身份证.png`

此外，`data/300.png`、`data/400.png` 和对应标准答案图可以用来观察“少量文字噪声与整体文字噪声不一致”的场景，以及不同调参组合对短信类样例的影响。当前策略已经支持 `400.png` 这种同图两处短信数字改动逐处输出。

对应测试预期是：

- 截图样例优先命中“时间篡改”
- 票据样例优先命中“数字篡改”
- 身份证样例需要覆盖多个重点区域，包括照片区，同时不过度误判姓名等正确文字
- 显式调高 `REPORT_CONFIDENCE_THRESHOLD` 后，截图、票据、300、400 样例仍应保留对应正确候选类型，且不应把与标准答案无重叠的错误候选留在最终可见结果中

## 当前代码特点

### 优点

- 入口简单，CLI 和 Python API 都能直接用
- 没有模型依赖，部署轻
- 对文档类图片做了比较明显的场景特化
- 测试直接绑定样例图，回归比较直观

### 当前结构上的主要感受

- `document_detector.py` 体积很大，集中了大量规则、特征和候选合并逻辑
- 文档规则、证件特化规则、区域滑窗逻辑目前都堆在一个文件里，后续维护成本会偏高
- `document_detector.py` 和 `detector.py` 都有 `visualize_detection`，但主入口实际走的是 `detector.py` 版本

## 一句话总结

这个项目本质上是在做“针对文档类图片的传统视觉篡改检测”，核心思路是“文本结构规则定位 + 局部异常检测 + ELA/噪声证据融合”。
